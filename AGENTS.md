# Second Law Agent Guide

## Project North Star

Build a small, playable Unity demo for **《转生公会：第二法则》** with a tight loop:

`冒险者协会 -> 接委托 -> LF2-style battle -> 奖励/升级 -> 委托人来信/回信`

Favor small, testable improvements over large rewrites. The user is learning Unity, so code and docs should remain beginner-readable.

## Required Reading Before Changes

- `Docs/AI_AGENT_DEV_LOG.md` for current implementation status.
- Notion beginner architecture doc: https://www.notion.so/3674dfd23cc281c89859d68c73d13f3f
- `README.md` for controls and current demo scope.

## Current Architecture

- Runtime boot: `SecondLawBootstrapper` creates `SecondLawGame`; scenes can be empty.
- Data is currently hard-coded in `DemoData.cs` for fast iteration.
- Guild UI is built with UI Toolkit (`GuildUiController`, `Assets/Resources/UI/Guild/GuildHub.uxml`, `GuildHub.uss`).
- Battle HUD is generated by `HudController`.
- Combat logic lives in `BattleController`, `Combatant`, `PlayerController`, `SlimeAI`, and `Projectile`.
- Character visuals use PNG spritesheets in `Assets/Resources/Art/Characters/`.
- Do not add the Aseprite importer yet unless the user explicitly asks; use exported PNG spritesheets for the current prototype.

## Development Rules

- Keep changes small and easy to explain.
- Preserve the current playable loop unless the user explicitly asks to replace it.
- Prefer readable C# over clever abstraction.
- Keep generated Unity cache folders out of git: `Library/`, `Temp/`, `Logs/`, `UserSettings/`.
- Use `Resources` only for current prototype convenience; plan to migrate to ScriptableObjects/Addressables later if assets grow.
- If a visual asset is missing, preserve a fallback so Play mode still works.

## Documentation Rule

Do not update docs for every small code/resource edit. Use the `doc-commit` skill when the user explicitly invokes it, or when the user explicitly asks for a commit and the pending change materially affects behavior, resources, workflow, architecture, build/deploy, or agent guidance.

When doc updates are needed before a commit:

1. Update `Docs/AI_AGENT_DEV_LOG.md` with the short status change, important files, and known gaps.
2. Update the Notion beginner architecture doc when behavior, controls, file responsibilities, assets, or workflow change.
3. Update README only when commands, user-facing scope, or sharing/deploy instructions change.
4. Mention the doc updates in the final response.

If a change is exploratory and will not be committed, do not update Notion unless the user asks.

## Verification Checklist

- Run `git diff --check`.
- In Unity, verify Play mode when possible:
  - Guild UI appears.
  - Language toggle still works.
  - Accept request enters battle.
  - Player can move, attack, jump, guard, shoot, and use combo skills.
  - Enemy can move, attack, take damage, and die.
  - Victory returns to guild and shows rewards/letter.
- If Unity cannot be run, say so clearly and list what was checked instead.

## Commit Style

- Commit small milestones with plain Chinese messages, for example:
  - `使用生成的公会背景`
  - `添加运行时语言切换`
  - `替换占位战斗角色图`
- Do not commit broken Play mode intentionally.
